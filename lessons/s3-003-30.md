# Enable S3 Versioning by Default on Any Bucket Holding Data You Don't Want to Lose

**Problem:** An object in S3 is overwritten by a bad write or deleted by accident, and there is no way to get the previous content back. By the time anyone notices, the only copy is gone.

**Root Cause:** S3 versioning is **off by default**. Without it, `PutObject` to an existing key replaces the object in place and `DeleteObject` removes it permanently — there is no prior version to recover. This bites both access patterns: Write-Once-Read-Many data (a stray overwrite clobbers the canonical copy) and Write-Many-Read-Few data (any one of the frequent writes can corrupt or truncate the object with no undo).

**Solution:** Default to enabling versioning on any bucket whose contents you care about, and declare it in IaC so it can't drift off. Pair it with a lifecycle rule that expires *noncurrent* versions to keep storage cost bounded:
```yaml
DataBucket:
  Type: AWS::S3::Bucket
  Properties:
    BucketName: !Sub 'myapp-data-${Environment}-${AWS::AccountId}'
    VersioningConfiguration:
      Status: Enabled
    LifecycleConfiguration:
      Rules:
        - Id: ExpireOldVersions
          Status: Enabled
          NoncurrentVersionExpiration:
            NoncurrentDays: 30
```
Recovery is then a matter of listing object versions (`aws s3api list-object-versions`) and copying or restoring the wanted `VersionId`. A bonus: retained prior versions can serve as the common-ancestor source for 3-way merges of concurrently-edited objects.

**Key Insight:** Make versioning the default, not the exception. The only good reason to leave it off is if it genuinely does not matter whether the data is deleted or overwritten — for everything else, the small storage cost (capped with a noncurrent-version lifecycle rule) buys you a rollback you'll eventually need.
