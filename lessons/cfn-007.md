# Deploy the API Gateway Stack Before the Lambdas Stack

**Problem:** Adding new API Gateway resources and the dependent Lambda methods in one pass fails because the lambdas stack can't import resource IDs that don't exist yet.

**Root Cause:** In a split-stack architecture where the API Gateway resources live in one stack and the Lambda integrations/methods live in another, the lambdas stack uses `Fn::ImportValue` to reference the API Gateway resource IDs. If you deploy the lambdas stack first, those exports don't exist yet.

**Solution:** When adding new API Gateway resources, deploy the **api-gateway stack FIRST**, then the **lambdas stack**. The lambdas stack imports the resource IDs the api-gateway stack exports, so the producer must exist before the consumer.

**Key Insight:** Cross-stack export/import creates a hard ordering dependency. Whenever you introduce a new export, deploy the exporting stack before any stack that imports it.
