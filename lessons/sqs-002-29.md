# A Crashing Lambda Can Bypass the SQS DLQ (maxReceiveCount Never Increments)

**Problem:** A single "poison" SQS message reprocessed over and over and **never landed in the dead-letter queue**, despite a redrive policy of `maxReceiveCount=3`. The job record stayed stuck at `processing`, the same expensive work (LLM calls, multi-page writes) re-ran every few minutes, the main queue showed the message permanently in-flight (`ApproximateNumberOfMessagesNotVisible=1`), and the DLQ stayed empty. The loop did not self-terminate after 3 attempts; it had to be killed by hand and would otherwise have run until the message retention period (default 4 days) expired — burning compute the entire time.

**Root Cause:** SQS DLQ redrive counts deliveries via `ApproximateReceiveCount`, which increments **once per genuine SQS receive** — and each genuine receive produces a new Lambda invocation with a **new** `RequestId`. A *hard crash* is not a normal function error-return: when the execution environment dies mid-invocation (OOM kill, segfault, runtime exit), Lambda re-initializes and **replays the same invocation in place** — same `RequestId`, preceded by a fresh `INIT_START` — without returning the message to the queue as a new receive. Because the receive count never advances, `maxReceiveCount` is never reached, the message never moves to the DLQ, and it loops until retention expires.

The CloudWatch Logs signature is distinctive — repeated, evenly-spaced:

```
INIT_START Runtime Version: ...
START RequestId: 26c34637-...   <-- attempt 1
  <app logs, then silence>
INIT_START Runtime Version: ...
START RequestId: 26c34637-...   <-- SAME id, attempt 2
  ...
```

with **no `END`/`REPORT` and no `Task timed out`** between the restarts. That distinguishes the three cases:
- **Clean exception** → function returns an error, message redelivered as a *new* receive (new RequestId), `ApproximateReceiveCount`++ → DLQ works as intended.
- **Timeout** → emits `Task timed out after N.NN s` **and** a `REPORT` line.
- **Hard crash / OOM** → no error log, no `REPORT`, just `INIT_START` + the *same* `START RequestId` again. OOM in particular is silent: the process is `SIGKILL`ed, so you never even see the `Max Memory Used` (it lives on the `REPORT` line that never gets written).

**Solution:** Do not rely on the DLQ as your only backstop for poison messages — bound retries explicitly:

- **Enable `ReportBatchItemFailures`** on the event source mapping and return failed message IDs from the handler, so a failure becomes an explicit, counted redelivery instead of an in-place replay.
- **Add an application-level attempt cap** independent of SQS: keep an attempt counter in your job store (e.g. DynamoDB) and short-circuit to a terminal `error` state at the very top of the handler once it exceeds N — *before* any expensive work. This converts a poison pill into a fast no-op that drains the message regardless of how SQS is counting receives.
- **Prevent the crash in the first place:** right-size Lambda memory (memory also scales CPU, so more memory shortens long sequential work and reduces OOM risk); cap per-invocation work (fewer items per message, or checkpoint-and-resume so a restart continues instead of redoing everything); or move long/heavy work to Fargate for unbounded memory and runtime.
- **Alarm on the signature:** alert on invocations that end without an `END`/`REPORT`, or on a job re-entering `processing` under the same `RequestId`, so an in-place poison loop surfaces immediately instead of silently burning compute.

**Key Insight:** `maxReceiveCount`/DLQ protects you against messages that make your function **return an error** — not against messages that make your function **die**. An OOM or runtime crash retries in place under a single delivery and never reaches the DLQ. If a single message can crash the worker, the DLQ is not a sufficient safety net; bound the retries at the application layer (attempt counter) or force counted redeliveries with `ReportBatchItemFailures`.
